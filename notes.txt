LOCALINK MVP — PITCH NOTES (Customer + Vendor Marketplace)
==========================================================

1) WHAT IS LOCALINK?
-------------------
Localink is a local marketplace platform that connects:
- Customers who need a service/product nearby
- Vendors (shops/service providers) who offer services/products locally

Think of it like a simple hybrid of:
- Discovery like Swiggy (browse nearby vendors)
- Service request flow like Urban Company (customer requests, vendor accepts)

This MVP is built to prove the full working flow, not just UI.


2) WHAT PROBLEM DOES IT SOLVE?
-----------------------------
- Customers currently struggle to find trustworthy, nearby vendors quickly.
- Vendors struggle to get visibility and structured incoming requests.

Localink provides:
1) Vendor discovery (list of vendors)
2) Vendor profile + products
3) One-click request creation by customer
4) Vendor dashboard to accept requests + manage products


3) CORE FEATURES IMPLEMENTED (MUST-HAVE MVP)
-------------------------------------------

A) Authentication + Roles
   - Signup/Login for both customer and vendor
   - Role-based access control:
     - Customer can create requests
     - Vendor can add products and accept requests

B) Vendor Discovery (Homepage)
   - Customer can fetch list of vendors:
     - name, category, rating, location
   - API: GET /vendors

C) Vendor Profile Page
   - Vendor details + products list
   - Customer can click “Request Service”
   - APIs:
     - GET /vendor/:id
     - GET /products/:vendorId

D) Product System (Vendor adds products)
   - Product fields: name, price, description, vendorId
   - Vendor can create product:
     - API: POST /product (vendor-only)

E) Request System (important flow)
   - Customer creates request -> Vendor accepts request
   - Request model:
     - customerId, vendorId, type("service"), status("pending"|"accepted")
   - APIs:
     - POST /request (customer-only)
     - PUT /request/:id (vendor-only accept)
     - GET /requests (both roles)
   - Logic:
     - Prevent accepting an already accepted request (returns 409)
     - Vendor can only accept requests that belong to them

F) Vendor Dashboard
   - Vendor can view incoming requests (GET /requests)
   - Accept request (PUT /request/:id)
   - Add product (POST /product)


4) END-TO-END DEMO STORY (WHAT TO SHOW TO SIR)
----------------------------------------------

Two sessions at the same time:
- Browser 1 (Normal): Login as CUSTOMER
- Browser 2 (Incognito): Login as VENDOR

Demo steps:
1) Vendor signs up as vendor (category + location required)
2) Vendor adds a product from vendor dashboard
3) Customer logs in and opens vendor profile -> sees that product
4) Customer clicks “Request Service” -> creates a request (pending)
5) Vendor dashboard shows the incoming request -> vendor clicks “Accept”
6) Request becomes accepted; trying to accept again shows proper error handling

Important note for pitching:
- Both sessions see the same data because everything is stored in MongoDB.
- Customer views vendor/products via APIs; vendor updates products/requests via APIs.
- Customer will see updates after refresh because customer page re-fetches data from backend.


5) TECH STACK AND WHY IT’S GOOD FOR MVP
---------------------------------------

Backend:
- Node.js + Express.js: fast API development, easy routing and middleware
- MongoDB + Mongoose: flexible schema, quick iteration for MVP
- JWT Authentication: stateless token-based auth for web apps

Frontend:
- React: component-based UI, easy routing and state management
- Simple styling: clean, modern look inspired by the given reference design


6) BACKEND ARCHITECTURE (THEORY)
-------------------------------
Backend is a REST API server. It does NOT render web pages.
It exposes endpoints that the frontend calls.

Flow:
Frontend -> calls API routes -> routes call controllers -> controllers read/write MongoDB models -> response to frontend

Folder structure:
backend/
  src/
    config/
    models/
    controllers/
    routes/
    middleware/
    utils/
    app.js
    server.js


7) BACKEND FILE-BY-FILE (WHAT EACH FILE DOES)
---------------------------------------------

backend/package.json
  - Defines backend dependencies and scripts.
  - "dev" runs nodemon so server restarts on changes.

backend/.env.example
  - Template env vars:
    - PORT, MONGODB_URI, JWT_SECRET, CLIENT_ORIGIN

backend/src/config/env.js
  - Loads environment variables using dotenv.
  - Provides defaults for local development.

backend/src/config/db.js
  - Connects to MongoDB using Mongoose.

backend/src/utils/http.js
  - A helper to send consistent JSON error responses:
    { error: { message, details? } }

backend/src/models/User.js
  - User schema for both roles (customer/vendor).
  - Vendor profile fields:
    - category, rating, location
  - Unique email for login/signup.

backend/src/models/Product.js
  - Product schema:
    - name, price, description, vendorId
  - vendorId links product to a vendor user.

backend/src/models/Request.js
  - Request schema:
    - customerId, vendorId, type(service), status(pending/accepted)
  - Indexing for vendor incoming requests.

backend/src/middleware/auth.js
  - authRequired:
    - Reads JWT from Authorization header.
    - Validates token.
    - Loads user from DB and attaches to req.user.
  - requireRole("customer"|"vendor"):
    - Blocks routes if role doesn’t match.

backend/src/controllers/authController.js
  - signup:
    - Validates role and required fields.
    - Hashes password using bcrypt.
    - Creates user in DB.
    - Returns JWT + user profile.
  - login:
    - Validates credentials.
    - Returns JWT + user profile.
  - me:
    - Returns current user from req.user.

backend/src/controllers/vendorController.js
  - listVendors:
    - Returns vendors list for discovery page.
  - getVendor:
    - Returns a single vendor profile by id.

backend/src/controllers/productController.js
  - getProductsByVendor:
    - Lists products for vendor profile page.
  - createProduct (vendor-only):
    - Creates product for logged-in vendor.

backend/src/controllers/requestController.js
  - createRequest (customer-only):
    - Creates a new request with status "pending".
  - listRequests:
    - If vendor -> show incoming requests
    - If customer -> show outgoing requests
  - acceptRequest (vendor-only):
    - Only accepts if request belongs to the vendor.
    - Prevents double accept (returns 409 if already accepted).

backend/src/routes/authRoutes.js
  - Routes:
    - POST /auth/signup
    - POST /auth/login
    - GET /auth/me

backend/src/routes/vendorRoutes.js
  - Routes:
    - GET /vendors
    - GET /vendor/:id

backend/src/routes/productRoutes.js
  - Routes:
    - GET /products/:vendorId
    - POST /product (vendor-only)

backend/src/routes/requestRoutes.js
  - Routes:
    - POST /request (customer-only)
    - PUT /request/:id (vendor-only accept)
    - GET /requests (role-scoped results)

backend/src/app.js
  - Creates express app.
  - Adds:
    - CORS (frontend allowed origin)
    - JSON body parser
    - Routes registration
    - 404 handler
    - Global error handler

backend/src/server.js
  - Connects to DB.
  - Starts listening on PORT.


8) FRONTEND ARCHITECTURE (THEORY)
---------------------------------
Frontend is a React app that:
- Handles the UI screens
- Stores auth token in localStorage
- Calls backend APIs to load/update real data

Folder structure:
frontend/
  src/
    app/
    pages/
    components/
    services/
    styles.css
    main.jsx


9) FRONTEND FILE-BY-FILE (WHAT EACH FILE DOES)
----------------------------------------------

frontend/package.json
  - Defines React + Vite dependencies and scripts.

frontend/vite.config.js
  - Vite development server configuration (runs on 5173).

frontend/.env.example
  - Defines VITE_API_URL (backend URL).

frontend/index.html
  - Root HTML shell where React mounts.

frontend/src/main.jsx
  - Entry point:
    - Mounts React app
    - Wraps with BrowserRouter
    - Wraps with AuthProvider

frontend/src/styles.css
  - Global styling:
    - Modern header, search bar, vendor cards, badges
    - Layout inspired by the reference image

frontend/src/services/api.js
  - API client wrapper:
    - Adds JSON headers
    - Adds Authorization Bearer token when provided
    - Throws friendly errors if response is not OK

frontend/src/app/auth/storage.js
  - localStorage helper:
    - loadAuth(), saveAuth(), clearAuth()

frontend/src/app/auth/AuthContext.jsx
  - Central auth state:
    - login(), signup(), logout()
    - Stores {token, user} in localStorage
    - Exposes token/user to the whole app

frontend/src/app/App.jsx
  - Routing:
    - /auth (login/signup)
    - /vendors (customer discovery)
    - /vendor/:id (vendor profile)
    - /vendor/dashboard (vendor dashboard)
  - Protected routing:
    - If not logged in -> redirected to /auth
  - Role-based default landing:
    - vendor -> dashboard
    - customer -> vendors

frontend/src/components/Layout.jsx
  - Top header UI:
    - Brand + location + search UI
    - Navigation links based on role
    - Logout button
  - Renders <Outlet /> to show pages below the header.

frontend/src/components/Field.jsx
  - Small reusable form field wrapper (label + input).

frontend/src/pages/AuthPage.jsx
  - UI for login/signup:
    - customer signup: name/email/password
    - vendor signup: plus category + location
  - Calls backend:
    - POST /auth/signup
    - POST /auth/login

frontend/src/pages/CustomerHomePage.jsx
  - Customer discovery page:
    - Calls GET /vendors
    - Displays vendor cards
    - Clicking vendor opens vendor profile page

frontend/src/pages/VendorProfilePage.jsx
  - Vendor profile page:
    - Calls GET /vendor/:id
    - Calls GET /products/:vendorId
    - Customer can create request:
      - POST /request

frontend/src/pages/VendorDashboardPage.jsx
  - Vendor dashboard:
    - Calls GET /requests to show incoming requests
    - Accept request:
      - PUT /request/:id
    - Add product:
      - POST /product


10) WHAT MAKES THIS “PRODUCTION-QUALITY MVP”?
--------------------------------------------
- Clean separation of concerns:
  routes -> controllers -> models
- JWT authentication + role-based authorization
- Proper status handling for requests:
  - prevents double acceptance
- Real database persistence:
  - vendor/product/request updates are stored in MongoDB
- Frontend talks to backend via a single API service layer
- UI is modern and pitch-ready while keeping scope minimal
